# お薬情報整理

歯科医院向けの、お薬手帳・薬袋写真から薬品情報を整理する Web アプリです。  
患者データをサーバーや DB に保存せず、その場で解析・印刷するステートレス構成です。

## 課題と解決

### 現場の課題

歯科の定期検診では、来院ごとに患者さんのお薬を確認しています。現状は紙のお薬手帳をコピーし、必要に応じて薬を調べる運用で、次のような負担があります。

- 高齢の患者さんが多く、服用薬の種類・変化が多い一方、ご本人も把握しきれていない
- マイナンバーの薬情報は参照できてもコピーできず、その日限りで残せない
- 3ヶ月ごとの検診で毎回確認するため手間が大きく、検診時間短縮（60分→45分）のネックの一つになっている

### このアプリでやったこと

形式の違うお薬情報（紙・画面・薬袋など）を写真から読み取り、**統一フォーマットの表**に整え、**前回との比較**と**薬の概要確認**、**印刷**までできるようにしました。確認作業を短縮しつつ、院内で使い回せる紙（＋QR）として残せます。

IT に苦手意識の強い職場でも使えるよう、**カメラで撮るだけで進められる手軽さ**を重視しています。

## できること

| 機能 | 内容 |
|------|------|
| **お薬表作成** | 写真（複数枚）から薬品名を抽出し、用途・歯科での注意を表形式で表示・A4印刷 |
| **前回比較** | 前回／今回の写真を比較し、増えた薬・消えた薬・継続中の薬を整理 |
| **保存なし QR** | 印刷用紙に薬一覧 QR を付与。比較画面では前回 QR を読み取って再利用可能 |
| **歯科注意の出し分け** | 院内リストに基づき、特記がある薬だけ注意を表示 |
| **印刷最適化** | 余白調整・白黒印刷・表の改ページなど、院内プリンタ向けのレイアウト |
| **簡易パスワード保護** | 院内共有パスワードでログイン。約30日間維持し、ログアウトも可能 |

患者番号は印刷用紙への反映用のみで、保存しません。

## 技術構成

- **Frontend / Backend:** Next.js（App Router）+ TypeScript + Tailwind CSS
- **AI:** Google Gemini（薬品抽出）
- **ホスティング:** Vercel（サーバーレス）
- **データ:** DB なし（リクエスト単位の使い捨て処理）

```
写真アップロード
  → 画像圧縮
  → /api/analyze または /api/compare
  → Gemini で抽出
  → 画面表示・A4印刷・QR
```

### なぜこの構成か

最速で院内実用化するため、学習目的ではなく、現場の制約に合う技術を選びました。

- **Next.js と Vercel**  
  タブレット向け UI（大きなボタン・撮影・印刷）と、API・公開をまとめて早く届けるために選んだ。

- **DB なし（その場限りの処理）**  
  患者情報をサーバーに残したくない、という方針に合わせた。扱うのは確認用の患者番号だけで、写真や薬データは保存しない。残るのは印刷した紙（と紙上の QR）だけにし、漏えいのリスクを小さくしている。

- **Google Gemini**  
  紙・スマホ画面・薬袋など、形の違う写真から薬品名を読む必要があるため。コストと速度のバランスが取れるモデルを使い、歯科で特に注意したい薬は院内リストでも出し分けている。

- **カメラで撮る入力**  
  IT に苦手意識の強い職場でも使えるよう、難しい操作を増やさず「撮る → 確認 → 印刷」で完結するようにした（PC では画像ファイルの選択にも対応）。

## 工夫・試行錯誤した点

お薬表作成では、速さと読み取り精度の両立が一番の課題でした。

写真は枚数が多いほど Gemini への送信・解析に時間がかかるため、画像圧縮や軽量モデル（gemini-3.1-flash-lite）で応答を速くする方針を取りました。一方で、圧縮を強くしすぎると字が潰れて薬品の漏れが増えることがあり、特に読みにくい写真を1枚だけ使うと比較で欠落が出やすい、という問題にも直面しました。

そこで、複数枚は軽めの圧縮で速さを優先し、1枚のときだけ解像度を上げて送るようにしました。モデルを重いものに上げるのではなく、入力画像の質で精度を補う判断です。あわせて、比較では前回→今回を順に解析する、読み込み中は「準備→送信→解析」と段階表示する、画像送信を multipart にする、Vercel の実行リージョンを東京にするなど、待ち時間の短縮と体感の改善も重ねています。

また AI の結果は完全ではない前提で、最終確認は人が行うこと、読みにくいときは別角度をもう1枚足すことなど、運用側の工夫もセットで考えています。

## 使ってみての改善（アップデート）
院内で実際に使用してみて、スタッフからフィードバックを貰い改善した内容です。

**問題1：個人情報をGeminiに送ることへの懸念と、対策の手間**
- Geminiに個人情報を送るのは心配
- 既存の対策は写真を撮影する際に名前などの個人情報を写さない（隠すなど）で面倒に感じる

**↪︎改善内容：写真確認＋任意の黒塗り**
- 撮影後にプレビューし、氏名などが写った部分だけ指で黒塗りしてから AI に送れるようにした。必須ではなく、不要ならすぐ OK で進められる。

**問題2：QRコードの読み取りができているか撮影中にわかりにくい**
- QRコード読み取りの撮影画面もQRコード用の枠がない通常のカメラで撮影、撮影後にQRコードを読み取る仕様だった為、撮影中に正しくQRコードを撮影できているかわからず困った

**↪︎改善内容：QR のライブ読み取り**
- 「撮ってあとで判定」から、枠内に合わせると読めたか分かるライブスキャンに変更し、院内タブレットでの手戻りを減らした。

## ローカル起動

```bash
git clone https://github.com/tachiya-kaoru/medication-check.git
cd medication-check
npm install
cp .env.local.example .env.local
```

`.env.local` に次を設定します。

```
GEMINI_API_KEY=your_api_key_here
SITE_PASSWORD=your_shared_password
```

- `GEMINI_API_KEY` … [Google AI Studio](https://aistudio.google.com/apikey) で発行（必須）
- `SITE_PASSWORD` … 院内共有パスワード（設定するとログイン保護が有効。未設定なら保護なし）

```bash
npm run dev
```

http://localhost:3000 を開きます（`SITE_PASSWORD` 設定時は `/login` へ誘導されます）。

| 画面 | URL |
|------|-----|
| お薬表作成 | `/` |
| 前回比較 | `/compare` |

## 環境変数

| 変数名 | 説明 |
|--------|------|
| `GEMINI_API_KEY` | Gemini API キー（必須） |
| `SITE_PASSWORD` | 院内アクセス用の共有パスワード（設定時のみログイン保護） |
| `AUTH_SECRET` | Cookie 署名用（任意。未設定時は `SITE_PASSWORD` を流用） |

Vercel では Project Settings → **Environments**（または Environment Variables）に同名で設定し、再デプロイします。  
`.env.local` はコミットしません。

## 設計上の方針

- **プライバシー優先:** 患者情報・画像を永続保存しない（マイナンバー情報を院内に溜めない制約にも合う）
- **院内運用前提:** タブレット撮影 → 確認 → 印刷の短い動線（操作を増やさない）
- **アクセス制限:** 公開 URL でも院外から自由に使えないよう、共有パスワードで保護（一度入ればしばらく再入力不要）
- **コスト意識:** 画像圧縮・軽量モデル利用で API コストと速度を抑える
- **拡張余地:** 電子版お薬手帳（JAHIS）QR 読取などは将来対応を想定
